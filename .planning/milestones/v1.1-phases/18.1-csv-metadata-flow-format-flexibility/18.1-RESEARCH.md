# Phase 18.1: CSV Metadata Flow & Format Flexibility - Research

**Researched:** 2026-04-12
**Domain:** CSV import pipeline, plan operation metadata passthrough, scan-to-plan bridging
**Confidence:** HIGH

## Summary

The CSV import system (`internal/planengine/csvimport.go`) has two related problems that block the `write_metadata` workflow from being usable in practice:

1. **Column name rigidity**: The parser requires exact column names `op_type`, `source_path`, `dest_path`. Real-world CSV exports from spreadsheet tools often produce variants like `Operation`, `Source Path`, `Type`, `Source`, `Destination`, etc. The existing case-insensitive handling helps but does not cover common naming variations or whitespace/underscore differences.

2. **No metadata passthrough**: When a user creates a CSV with `write_metadata` rows, they cannot supply metadata fields (title, author, narrator, genre, year, series, ASIN) in the CSV. The executor's `resolveBookMetadata` falls back to DB lookup and file extraction, but for books flagged as `missing_metadata`, these lookups return empty/incomplete data -- which is precisely why the user is trying to supply metadata via CSV in the first place.

3. **Bridge gap (INT-BRIDGE-01)**: The `missing_metadata` scan issue type is not in `actionableTypes` in `bridge.go`, so the scan-to-plan flow cannot auto-create `write_metadata` plans. Users are forced to use CSV import for `write_metadata` plans -- but CSV import itself cannot carry metadata.

**Primary recommendation:** Add a `metadata` JSON column to `plan_operations`, extend CSV import to accept metadata columns (title, author, narrator, etc.) with flexible column name matching, update the executor's `write_metadata` case to prefer operation-embedded metadata over DB/file lookups, and bridge `missing_metadata` to `write_metadata` in `actionableTypes`.

### Pattern: Column Name Aliasing

The CSV import should normalize column headers to canonical names using an alias map. This is a standard pattern for user-facing CSV parsers.

```go
// columnAliases maps common real-world header variants to canonical names.
var columnAliases = map[string]string{
    "op_type":     "op_type",
    "type":        "op_type",
    "operation":   "op_type",
    "action":      "op_type",
    "source_path": "source_path",
    "source":      "source_path",
    "path":        "source_path",
    "src":         "source_path",
    "dest_path":   "dest_path",
    "destination": "dest_path",
    "dest":        "dest_path",
    "target":      "dest_path",
    "title":       "title",
    "author":      "author",
    "narrator":    "narrator",
    "genre":       "genre",
    "year":        "year",
    "series":      "series",
    "asin":        "asin",
}
```

Headers should be lowercased, trimmed, and have spaces/hyphens normalized to underscores before alias lookup.

### Pattern: Metadata JSON in Operations

The `plan_operations` table gets a `metadata` TEXT column (default empty string) that stores a JSON blob. This is the standard SQLite approach for semi-structured data that varies by operation type.

```go
type PlanOperation struct {
    // ... existing fields ...
    Metadata     string // JSON blob, empty for non-metadata ops
}
```

For `write_metadata` operations, the JSON encodes the fields from `metadata.BookMetadata` plus ASIN:

```json
{
    "title": "The Great Book",
    "author": "Jane Author",
    "narrator": "Bob Reader",
    "genre": "Fiction",
    "year": 2023,
    "series": "My Series",
    "asin": "B0XXXXXXXX"
}
```

### Anti-Patterns to Avoid

- **Separate metadata table for operations:** Over-engineering. A JSON column is simpler, avoids joins, and metadata is always read/written with its operation.
- **Encoding metadata in dest_path:** Hack that breaks the data model. dest_path has semantic meaning for move/split operations.
- **Making metadata columns nullable on plan_operations:** Many nullable columns create sparse rows. JSON blob is cleaner.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Column name normalization | Custom per-column if/else chains | Alias map with normalize function | Maintainable, extensible, testable |
| JSON serialization | Manual string building | encoding/json.Marshal/Unmarshal | Go stdlib, handles escaping correctly |

## Common Pitfalls

### Pitfall 1: Migration Breaking Existing Data
**What goes wrong:** Adding a NOT NULL column without a default breaks existing rows.
**Why it happens:** SQLite requires defaults for new NOT NULL columns on tables with data.
**How to avoid:** Use `TEXT NOT NULL DEFAULT ''` for the metadata column. Empty string means "no metadata provided."
**Warning signs:** Migration fails on `ALTER TABLE` with "cannot add NOT NULL column without default."

### Pitfall 2: Column Alias Ambiguity
**What goes wrong:** Multiple CSV columns map to the same canonical name (e.g., both "source" and "source_path" present).
**Why it happens:** User exports from different tools, or copies columns.
**How to avoid:** If multiple columns resolve to the same canonical name, use the first match and log a warning in CSVImportResult. Alternatively, reject as ambiguous.
**Warning signs:** Silent data loss from wrong column being used.

### Pitfall 3: Metadata JSON Encoding Inconsistency
**What goes wrong:** CSV metadata stored as JSON doesn't match what `resolveBookMetadata` returns, causing different write_metadata behavior depending on source.
**Why it happens:** Two code paths produce metadata in different shapes.
**How to avoid:** Both paths should produce the same intermediate type (`metadata.BookMetadata` + ASIN string) that feeds into `BuildABSMetadata`. The JSON in the operation should deserialize to `BookMetadata`.
**Warning signs:** Metadata sidecars have different field sets depending on whether they came from CSV or DB.

### Pitfall 4: Year as String vs Int in CSV
**What goes wrong:** CSV always produces strings. `BookMetadata.Year` is `int`. Parsing fails silently or produces 0.
**Why it happens:** CSV has no type information.
**How to avoid:** Parse year with `strconv.Atoi`, treat empty/non-numeric as 0 (matching existing behavior where 0 year produces empty publishedYear in ABS metadata).
**Warning signs:** All years showing as empty in metadata.json files.

### Pitfall 5: Bridge Creates write_metadata Without Metadata
**What goes wrong:** Adding `missing_metadata -> write_metadata` to bridge's `actionableTypes` creates operations that only have `source_path` and no metadata. The executor falls back to `resolveBookMetadata`, which may return empty data.
**Why it happens:** The bridge doesn't have metadata to embed -- it only knows the path and issue type.
**How to avoid:** This is acceptable behavior. The bridge-created write_metadata ops use the existing fallback chain (DB lookup, file extraction). If the book IS in the DB with metadata (common for Audible-downloaded books), the fallback works. For truly metadata-less books, the bridge at least generates a plan the user can review/augment via CSV.
**Warning signs:** Empty metadata.json files after applying bridge-generated plans.

### CSV Import: Column Normalization

```go
// normalizeHeader converts a raw CSV header to its canonical form.
func normalizeHeader(raw string) string {
    s := strings.ToLower(strings.TrimSpace(raw))
    s = strings.ReplaceAll(s, " ", "_")
    s = strings.ReplaceAll(s, "-", "_")
    if canonical, ok := columnAliases[s]; ok {
        return canonical
    }
    return s // return as-is for metadata columns
}
```

### CSV Import: Metadata Extraction per Row

```go
// metadataColumns lists the CSV columns that map to BookMetadata fields.
var metadataColumns = []string{"title", "author", "narrator", "genre", "year", "series", "asin"}

func extractMetadataJSON(colIndex map[string]int, record []string) string {
    m := make(map[string]interface{})
    for _, col := range metadataColumns {
        idx, ok := colIndex[col]
        if !ok || idx >= len(record) {
            continue
        }
        val := strings.TrimSpace(record[idx])
        if val == "" {
            continue
        }
        if col == "year" {
            if y, err := strconv.Atoi(val); err == nil {
                m["year"] = y
            }
            continue
        }
        m[col] = val
    }
    if len(m) == 0 {
        return ""
    }
    data, _ := json.Marshal(m)
    return string(data)
}
```

### Executor: Prefer Operation Metadata

case "write_metadata":

## Open Questions

1. **Column alias collision policy**
   - What we know: A CSV could have both "source" and "source_path" columns.
   - What's unclear: Should we reject ambiguous CSVs or take first match?
   - Recommendation: Reject with a clear error listing the conflicting columns. Fail-loud is safer for data operations.

2. **Should bridge-created write_metadata ops include scan-extracted metadata?**
   - What we know: The bridge has access to the issue path but not metadata. The deep scanner already extracted metadata (or failed to) when creating the issue.
   - What's unclear: Whether to do a metadata extraction pass during bridge plan creation.
   - Recommendation: No. Keep the bridge simple. The executor's `resolveBookMetadata` fallback handles this. Users who need to override use CSV import.
